### 5.3. TDD (テスト駆動開発) の計画

本タスクはドキュメントの変更であり、既存のコードロジックへの影響が限定的であるため、主に以下のテスト観点に焦点を当てます。

1.  **単体テスト (Unit Test)**:
    *   **テスト対象**: `README.md`の内容を読み込み、指定された注記を挿入し、結果の文字列を返すユーティリティ関数（仮称: `insert_note_to_readme`）。
    *   **テストケース**: 
        *   **正常系**: 
            *   空の`README.md`に注記を挿入した場合、注記のみが追加されること。
            *   タイトルのみの`README.md`に注記を挿入した場合、タイトルの直後に注記が追加されること。
            *   タイトルと既存の本文がある`README.md`に注記を挿入した場合、適切な位置（例: 最初の段落の直後）に注記が追加され、他の内容は変更されないこと。
            *   複数行の`README.md`に注記を挿入した場合、指定された位置に1行として追加されること。
        *   **異常系**: 
            *   （該当なし、ファイル読み書きはOSレベルで処理されるため）
        *   **境界値**: 
            *   `README.md`が非常に短い場合（1行、数行）でも正しく挿入されること。
        *   **エッジケース**: 
            *   `README.md`に特殊文字やMarkdown記法が多数含まれていても、それらが壊れずに注記が挿入されること。

2.  **結合テスト (Integration Test)**:
    *   **テスト対象**: ファイルシステム上での`README.md`の読み書きと、注記挿入処理の結合。
    *   **テストケース**: 
        *   **正常系**: 
            *   一時ファイルとして`README.md`を作成し、処理を実行後、ファイルの内容が期待通りに更新されていることを確認する。
            *   処理実行後、元の`README.md`以外のファイルが変更されていないことを確認する。
        *   **異常系**: 
            *   `README.md`ファイルが存在しない場合に、エラーが適切にハンドリングされること（ただし、本タスクではファイルが存在することを前提とする）。
            *   `README.md`ファイルへの書き込み権限がない場合に、エラーが適切にハンドリングされること。

3.  **E2Eテスト (End-to-End Test)**:
    *   **テスト対象**: CI/CDパイプライン全体。
    *   **テストケース**: 
        *   **正常系**: 
            *   変更をコミットし、PRを作成した場合、CI/CDパイプラインが正常に実行され、すべての品質ゲート（Formatter, Linter, 静的解析, テスト, ビルド）をパスすること。
            *   PRがマージされた後、`main`ブランチの`README.md`が期待通りの内容になっていること。
